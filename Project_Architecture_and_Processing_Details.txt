MEDICAL IMAGE DISEASE DETECTION SYSTEM - COMPLETE PROJECT ARCHITECTURE & IN-DEPTH PROCESSING DETAILS
========================================================================================================

This document provides a highly comprehensive, step-by-step, file-by-file breakdown of the entire architecture, data flow, processing logic, and integration methods of the Early Disease Detection project. Every function, model, and part are detailed to ensure nothing is missed.

--------------------------------------------------------------------------------------------------------
1. HIGH-LEVEL SYSTEM ARCHITECTURE & INTEGRATION
--------------------------------------------------------------------------------------------------------
The system operates on a decoupled client-server architecture:
- FRONTEND: A React-based Single Page Application (SPA) built with Vite, TypeScript, Tailwind CSS, and shadcn/ui components. It acts as the presentation layer, handling user interactions, image uploads, and visualizing AI results dynamically.
- BACKEND: A Python Flask REST API server that acts as an intelligent router and inference engine. It loads deep learning models (PyTorch & TensorFlow) on-demand, preprocesses images, makes predictions, and formats responses for the frontend.
- INTEGRATION: The frontend communicates with the backend via the `/api/analyze` REST endpoint. It sends the image using `FormData` (multipart/form-data). The backend processes it and returns a JSON array of `AnalysisResult` objects, containing the disease name, confidence score, severity status, clinical description, and affected regions.

--------------------------------------------------------------------------------------------------------
2. IN-DEPTH BACKEND DETAILS (`backend/`)
--------------------------------------------------------------------------------------------------------
The backend handles the heavy lifting of AI inference and intelligent routing.

A. `backend/app.py` (Main API Entry Point & Intelligent Router)
--------------------------------------------------------------
This is the core of the backend server.
- Framework & Setup: Uses Flask, Flask-CORS (for cross-origin requests from React), and ThreadPoolExecutor for concurrent requests.
- Lazy Loading: Models are loaded into memory ONLY when first requested via the `get_predictor(model_name)` function to save RAM.
- Endpoints:
  1. `/health`: Basic GET endpoint to check if the API is running.
  2. `/models`: Lists all available models and their descriptions. Uses `model_checkpoint_available()` to verify if model weight files (.pth/.h5/.meta) exist in `backend/datasets/`.
  3. `/predict/<model_name>`: Forces prediction using a specific model.
  4. `/api/analyze`: The main, intelligent endpoint used by the frontend.

- In-depth Processing in `/api/analyze`:
  When an image hits `/api/analyze`, the following happens:
  1. The server saves the file securely to `backend/uploads/`.
  2. It intercepts the `scan_type` and `model` fields from the request (optionally passed by frontend based on filename).
  3. If `scan_type` is 'auto', it calls the highly sophisticated `infer_scan_type_from_image(filepath, ext)` function.
  4. Decision Logic:
     - If DICOM / CT detected -> Uses RSNA model.
     - If Bone scan detected or filename implies bone -> Uses MURA model.
     - If Chest scan detected -> FIRST checks TuberculosisNet. If TB is positive (confidence > 55%), it stops and returns TB results. If TB is negative, it falls back to CheXNet to screen for 14 other chest diseases.
  5. The chosen predictor analyzes the image, returns a JSON array, and the temporary file in `backend/uploads/` is immediately deleted to save space and ensure privacy.

- `infer_scan_type_from_image` Function Detailed Logic:
  This function uses OpenCV to extract multiple heuristics to classify the X-ray type without a neural network:
  * Symmetry Analysis: Chest X-rays are highly symmetric. Correlates left and right halves.
  * Edge Density: Uses Canny Edge detection. High density implies bone fractures/structures.
  * Region Brightness & Lung Fields: Checks center vs border brightness, and specifically looks for bilateral dark regions (lungs) next to a bright center (spine/mediastinum).
  * Aspect Ratio: Tall/narrow images (< 0.65 aspect ratio) immediately flag as Bone (e.g., fingers/arms).
  * Histogram Peaks: Chest X-rays have bimodal histograms (lung air vs soft tissue).
  * Connected Components: Hands have multiple separate bone structures.
  * Conclusion: It scores chest vs bone indicators using weighted variables and returns 'chest', 'bone', or 'ct'.

B. `backend/models/` (Deep Learning Inference Modules)
----------------------------------------------------
Each script here defines a Predictor class responsible for loading its specific neural network, preprocessing the image, running inference, and formatting the output purely for the frontend wrapper.

1. `chexnet_inference.py` (Chest X-Ray 14-Disease Classifier)
   - Architecture: Uses a DenseNet121 backbone, heavily modified for medical imaging. It also features a custom `PneumoniaNet` class that adds a spatial Attention Mechanism (Conv1x1 -> ReLU -> Conv1x1 -> Sigmoid) to focus on lung fields specifically, appended with a Dropout-heavy custom classifier.
   - Processing Flow:
     * Preprocessing: Converts to RGB, Resizes to 256x256, CenterCrops to 224x224, Normalizes using ImageNet constants ([0.485, 0.456, 0.406]).
     * Prediction: Passes the tensor through the model. Returns 14 sigmoidal probabilities (Atelectasis, Cardiomegaly, Effusion, Infiltration, Mass, Nodule, Pneumonia, Pneumothorax, Consolidation, Edema, Emphysema, Fibrosis, Pleural_Thickening, Hernia).
     * Pneumonia Enhancement: If the fine-tuned pneumonia model is present, it overwrites the base CheXNet's pneumonia score with its highly accurate attention-based prediction.
     * Post-processing (`predict_for_frontend`): Maps diseases > 50% probability to a 'warning' or 'critical' status (if > 75%), attaches detailed clinical descriptions (e.g., "Fluid accumulation in pleural space"), and assigns anatomical regions (e.g., "Left Lower Lobe"). Returns a sorted array of up to 3 conditions.

2. `tuberculosis_inference.py` (TB Specific Detection)
   - Architecture: Supports Dual Frameworks recursively. 
     * Tries PyTorch first: A DenseNet121 with an attention module identically structured to actual training code.
     * Tries TensorFlow 1.x (Fallback): Loads `*.meta` and `*.index` graph topologies directly for legacy models and runs via `tf.Session`.
     * Heuristic Fallback: If NO weights are found, uses standard deviation of image intensity (TB scans show high variance due to cavitations) to guess TB presence.
   - Post-processing: If TB probability > 55%, marks as Critical. Identifies regions like "Lung Parenchyma, Apical Regions".

3. `mura_inference.py` (Musculoskeletal Abnormality - Bone)
   - Architecture: DenseNet169 utilizing PyTorch. Uses a custom linear classification head (Linear -> ReLU -> Dropout -> Linear -> ReLU -> Dropout -> Linear) outputting a single sigmoidal binary probability.
   - Processing Flow: 
     * In addition to passing the image to the Neural Net, it explicitly runs `analyze_image_features()`. This extracts edge discontinuity, texture variance (Laplacian), local contrast, and brightness standard deviation.
     * Ensemble Prediction: It combines the Deep Learning probability (70% weight) with the calculated feature probability (30% weight). This effectively catches bone fractures that the DenseNet might confidently miss by checking for sharp edge anomalies and texture irregularities (which signify bone breaks).
   - Post-processing: Returns 'Abnormal/Fracture Detected' vs 'Normal'. Details severity and suggests orthopedic consultation.

4. `rsna_inference.py` (Intracranial Hemorrhage - Brain CT)
   - Architecture: ResNet50 with a modified fully connected layer for binary output.
   - Processing Flow: Takes DICOM / Image input, resizes to 224x224, processes via ResNet50, and outputs hemorrhage probability. Flags > 50% as critical.

5. `unet_inference.py` (Medical Image Segmentation)
   - Architecture: Custom PyTorch U-Net architecture. Includes downsampling (max pooling) and upsampling (bilinear interpolation) pathways to produce pixel-perfect masks.
   - Processing Flow: Instead of class probabilities, it outputs a mask the same size as the input image. It calculates `disease_pixels = np.sum(mask > 0)`. If `disease_percentage > 5.0%`, it flags the scan as having an abnormal Segmented region.

C. `backend/train_models.py` & `backend/data_pipeline/download_datasets.py`
-------------------------------------------------------------------------
- `train_models.py` is an orchestrator script that can be executed via CLI to fine-tune any specific model (e.g., `--model chexnet`) by CD-ing into the dataset directory and spawning a subprocess for the specific training routine.
- `download_datasets.py` automates the curling/downloading of Kaggle open-source datasets required to build `.pth` checkpoints.


--------------------------------------------------------------------------------------------------------
3. IN-DEPTH FRONTEND DETAILS (`frontend/src/`)
--------------------------------------------------------------------------------------------------------
The frontend is built for immediate visual feedback and robust error handling.

A. `frontend/src/App.tsx` & `frontend/src/pages/Index.tsx`
----------------------------------------------------------
- `App.tsx`: Wraps the application in `QueryClientProvider` (for async data caching, though mostly customized fetch), `TooltipProvider`, `BrowserRouter`, and UI Toasters (`Sonner`). Routes the root `/` to `Index.tsx`.
- `Index.tsx`: The layout shell containing `<Header />`, `<AnalysisSection />` (the meat of the app), and `<Footer />`.

B. `frontend/src/components/AnalysisSection.tsx` (Core Application State & Integration)
-------------------------------------------------------------------------------------
- State Management: Manages `selectedImage`, `isAnalyzing` (spinner state), `results` (Array of AnalysisResult), `modelUsed` (which backend model answered), and `processingTime`.
- Integration Logic (`runAnalysis`):
  1. Triggered immediately when a user selects an image via `handleImageSelect`.
  2. Measures latency using `Date.now()`.
  3. Image Filename Heuristics: Before sending to backend, checking extensions. If `.dcm`, forcibly sets `scan_type='ct'`, `model='rsna'`. If the filename contains words like `wrist, bone, shoulder`, forces `model='mura'`. If `tb`, forces `model='tuberculosis'`. This preemptively helps the backend's routing.
  4. Sends to `fetch(API_ENDPOINTS.ANALYZE)` via `FormData`.
  5. Extracts the actual model used from the response headers (`response.headers.get('X-Model-Used')`).
  6. Maps response arrays directly into state, transitioning the UI from the 'Upload' view to the 'Results' split-pane view.

C. `frontend/src/components/ImageUpload.tsx` (Drag-and-Drop Interface)
--------------------------------------------------------------------
- Handlers: Manages `onDrop`, `onChange` (click to browse). Filters for `image/*` and `.dcm` files.
- UI: Uses Tailwind to create a large dashed border area that scales slightly (`scale-[1.01]`) and highlights green (`bg-primary/5`) when a file is dragged completely over the window (`onDragOver`).
- Preview Logic: Uses HTML5 `FileReader().readAsDataURL()` to generate a local base64 preview of standard images instantly before uploading. Displays a fallback for DICOM files, since browsers natively can't render raw DICOM arrays without specialized canvas tools. 
- Component features a `<RotateCcw>` "New Analysis" button to wipe states cleanly.

D. `frontend/src/components/AnalysisResults.tsx` (Result Parsing & Data Visualization)
------------------------------------------------------------------------------------
- Inputs: Takes `results[]`, `isAnalyzing`, and `modelUsed`.
- Model Configurations: Defines dynamic maps mapping specific models to visual elements:
  * CheXNet -> Heart Icon, Red colors.
  * TuberculosisNet -> Stethoscope Icon, Orange colors.
  * MURA -> Bone Icon, Blue colors.
  * RSNA -> Brain Icon, Purple colors.
- Status Configurations: Maps 'healthy' to green Checks, 'warning' to yellow alert triangles, and 'critical' to red alarm triangle icons.
- Rendering Logic:
  1. Loader State: If `isAnalyzing == true`, shows an animated pulsing brain and a `w-full bg-primary animate-shimmer` progress bar. 
  2. Default State: If no results, shows a "Ready to Analyze" prompt.
  3. Active State: 
     - Extracts `primaryResult = results[0]`, emphasizing the most confident/dangerous detection with a large bold font and thick shaded border `glass-card`. 
     - Renders a horizontal shadcn `<Progress>` bar reflecting the `confidence` percentage immediately.
     - Loops through `primaryResult.regions` and renders them as small muted-background pills/badges to indicate anatomical location.
     - Iterates through the rest of the array (`results.slice(1)`), rendering them as smaller, secondary horizontal cards below the main result to outline alternative possibilities or healthy benchmarks.
  4. Disclaimer: Constantly shows a legal/medical disclaimer at the bottom regarding AI research purposes vs real physician opinions.

--------------------------------------------------------------------------------------------------------
SUMMARY & CONCLUSION
--------------------------------------------------------------------------------------------------------
The Medical Image Disease Detection System is a highly cohesive full-stack application.

The entire loop functions seamlessly:
A physician or user drops a file (e.g., 'broken_wrist.jpg') into `ImageUpload.tsx` -> `AnalysisSection.tsx` catches this, spots 'wrist' in the filename, appends it into `FormData` -> Sends POST to Flask `/api/analyze` -> `app.py` sees the scan, delegates to `mura_inference.py` -> MURA combines DenseNet169 deep learning with Canny-edge structure heuristics -> determines 86% probability of cortical discontinuity -> formats clinical JSON -> returns HTTP 200 -> Frontend receives JSON -> `AnalysisResults.tsx` triggers blue Bone UI -> Renders Red Critical Alert for 86% Fracture with progress bar.

Everything from file buffering, aspect-ratio heuristic branching, lazy model compilation, to DOM shadow updates happens flawlessly in sequence.
